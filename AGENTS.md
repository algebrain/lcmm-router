# Руководство по репозиторию

## Структура проекта и организация модулей

`src/` содержит основной код библиотеки
`test/` содержит тесты
`docs/` содержит документацию для пользователя библиотеки: архитектурные заметки, API‑документацию и требования к поведению. Эти документы могут содержать детали реализации, но лишь в том случае если это необходимо для разъяснения каких-либо вопросов, важных для пользователя библиотеки
Когда вы просите создать план работ, документ плана должен быть размещён в `.plans/` и начинаться с `YYYY-MM-DD_`.  
Когда вы просите сохранить ответ, он должен быть размещён в `.answers/` и начинаться с `YYYY-MM-DD_`.  
Обе директории перечислены в `.gitignore` и не должны коммититься.  
Если нужно создать нестандартные файлы или папки, которые не должны попадать в git, их имена можно заканчивать на `.local` или `.local.*` — такие файлы и папки будут отсеяны через `.gitignore`.  
Используйте `README.md` как точку входа, затем следуйте ссылкам в `docs/` для подробностей.

## Команды сборки, тестирования и разработки

Это библиотека на Clojure без отдельных скриптов сборки/запуска. Основной рабочий процесс — тесты:

- `clj -M:test`  
  Запускает полный набор тестов через Kaocha, согласно `tests.edn`.
- `bb test.bb`  
  Хелпер, запускающий тесты с documentation reporter. Запускает линтинг, тесты и форматирование, в этом порядке.
  Поддерживает таймаут: `--timeout-min=<N>` или `--timeout-ms=<N>` (по умолчанию 5 минут).

Если нужен REPL, используйте предпочитаемый `clj`‑запуск и загрузите требуемый namespace

## Файловые операции в Windows

Все файлы должны сохраняться в UTF‑8 без BOM. BOM запрещён.  
Для файловых операций (чтение/запись) использовать Python вместо PowerShell, чтобы избежать проблем с кодировкой.
Любое чтение/просмотр и запись файлов выполнять через Python, не через PowerShell.
Примеры корректных файловых операций через Python (PowerShell не использовать):
  - Прочитать файл (utf-8): @'
from pathlib import Path
print(Path('AGENTS.md').read_text(encoding='utf-8'))
'@ | python -
  - Прочитать файл с возможным BOM: @'
from pathlib import Path
print(Path('AGENTS.md').read_text(encoding='utf-8-sig'))
'@ | python -
  - Записать файл (utf-8 без BOM): @'
from pathlib import Path
Path('test.bb').write_text("(println \"ok\")\n", encoding='utf-8')
'@ | python -

## Стиль кода и соглашения об именовании

Коммуникация предпочтительна на русском; комментарии в коде — на английском.  
Следуйте идиоматическому форматированию Clojure: отступ 2 пробела, выравнивание карт и биндингов, держите чистые функции небольшими.  
Имена namespace — kebab-case (например `event-bus`), имена файлов — с подчёркиваниями (например `event_bus.clj`).  
Ключевые слова используют формат `:domain/name` или `:module/name` (например, `:test/event`).

## Руководство по тестированию

ВНИМАНИЕ! Если каким-то образом в коде обнаружена ошибка, а тесты зеленые, то прежде чем вносить исправления необходимо добавить тест, который обнаруживает эту ошибку (то есть заканчивается неуспешно), и лишь потом восстановить успешное прохождение тестов путем исправления этой ошибки

Запускайте тесты через `bb test.bb`.
Тесты выполняются через Kaocha.
`bb test.bb` запускает линтинг, тесты и форматирование (в этом порядке).  
Тестовые файлы лежат в `test/` и используют namespace с суффиксом `*-test` (например, `event-bus-test`).  
Явной цели по покрытию нет; добавляйте тесты для нового поведения и краевых случаев, особенно вокруг причинности и валидации схем.

Цикл бенчмарков начинается с тройного бипа — его делать только перед запуском первого бенчмарка в цикле. Перед каждым последующим запуском бенчмарков в том же цикле — одинарный бип.
Перед запуском тестов и бенчмарков выводить текущие время в 24‑часовом формате (например, `STARTED AT 16:25`).
Цикл тестирование‑правки начинать с тройного бипа — его делать только перед запуском первого теста в цикле. Перед каждым последующим запуском тестов в том же цикле — одинарный бип.

ОЧЕНЬ ВАЖНО: при добавлении новой функциональности изменения в документации производятся ТОЛЬКО ПОСЛЕ того как тесты этой новой функциональности стали зелеными.

Правильный бип в PowerShell (замена Write-Host): `[console]::Beep(800,200)`; тройной бип: `[console]::Beep(800,200); [console]::Beep(1000,200); [console]::Beep(1200,200)`.

## Рекомендации по коммитам и pull request

История коммитов содержит короткие описательные сообщения (например, "README fix", "significant changes in code and documentation").  
Строгого стандарта нет, но коммиты должны быть краткими и сфокусированными на одном изменении.

Для pull request включайте:

- Краткое описание изменений поведения.
- Команды тестов и их результаты.
- Обновления в `README.md`, `docs/` при изменении поведения или API.

## Точки обновления документации

Если вы меняете публичный API или поведение, обновите `docs` (и английские варианты, если есть) и синхронизируйте `README.md`.
