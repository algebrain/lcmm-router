# Реализация компонента Router для модульного монолита

## 1. Цель

Создать централизованный компонент управления маршрутами (`Router`), который позволяет независимым модулям системы регистрировать свои эндпойнты (`handlers`) без прямой зависимости друг от друга. Результатом работы роутера должен быть стандартный `Ring-handler`.

## 2. Архитектурные принципы

- **Inversion of Control (IoC):** Модули сами объявляют свои маршруты, используя предоставленный им экземпляр роутера.
- **Thread Safety:** Регистрация маршрутов должна быть потокобезопасной, так как модули могут инициализироваться параллельно.
- **Ring Compliance:** Все обработчики (`handlers`) должны соответствовать спецификации Ring.
- **Low Coupling:** Роутер не должен знать о внутренней логике модулей.

## 3. Ключевые функции (Интерфейс)

Роутер должен предоставлять следующий программный интерфейс (API):

### А. Регистрация маршрутов

Функция (или метод), принимающая:

- **HTTP метод:** `:get`, `:post`, `:put`, `:delete` и т.д.
- **Путь:** Строка или шаблон, например `"/users/:id"`.
- **Handler-функция:** Функция, принимающая `request` и возвращающая `response`.
- **(Опционально) Map с опциями,** который может содержать:
  - `:name` — имя маршрута для генерации ссылок.
  - `:middleware` — вектор middleware для конкретного маршрута.

### Б. Компиляция (Build)

Функция, которая превращает накопленные данные о маршрутах в финальный `ring-handler`. После вызова этой функции изменения в роутер вноситься не должны (immutable state после инициализации).

## 4. Рекомендуемая структура данных и библиотека

Рекомендуется использовать библиотеку **Reitit**, так как она является стандартом де-факто в Clojure, обладает высокой производительностью и поддерживает разделение описания маршрутов и их компиляции.

- **Хранилище:** Использовать `atom`, содержащий `map`, где ключами являются пути, а значениями — `map`'ы с описанием маршрутов (`{:name :... :get ...}`). Это решает проблему дублирования путей.
- **Протокол (опционально):** Описать протокол `IRouter` для удобства тестирования и замены.

## 5. Пример реализации (для исполнителя)

```clojure
(ns component.router
  (:require [reitit.ring :as ring]))

(defprotocol IRouter
  (add-route! [this method path handler] [this method path handler opts]
    "Регистрирует новый эндпойнт в роутере.
     `opts` - map с ключами :name, :middleware и др. опциями Reitit."))

(defrecord Router [routes-atom]
  IRouter
  (add-route! [this method path handler]
    (add-route! this method path handler {}))
  
  (add-route! [this method path handler opts]
    (let [route-data (assoc opts method handler)]
      (swap! routes-atom
             (fn [routes]
               (update routes path merge route-data))))))

(defn make-router
  "Создает новый экземпляр роутера."
  []
  (->Router (atom {})))

(defn as-ring-handler
  "Компилирует маршруты в Ring-совместимый обработчик.
   Принимает опциональный map с глобальными Reitit-опциями (например, middleware)."
  ([router]
   (as-ring-handler router {}))
  ([router opts]
   (let [final-routes (->> @(:routes-atom router)
                           (mapv (fn [[path data]] (into [path] (vals data)))))]
     (ring/ring-handler
       (ring/router final-routes opts)
       (ring/create-default-handler)))))
```

## 6. Жизненный цикл в системе

1.  **Initialization:** В главном файле (`system/main`) создается экземпляр `(make-router)`.
2.  **Registration:** Экземпляр передается в функции инициализации каждого модуля:
    ```clojure
    (module.orders/init! router)
    (module.users/init! router)
    ```
3.  **Startup:** Управляющий модуль вызывает `(as-ring-handler router)` и передает результат в `http-kit/run-server`.

## 7. Требования к реализации

- **Поддержка Middleware:** Роутер должен позволять добавлять Middleware как глобально (через опции `as-ring-handler`), так и на уровне конкретного маршрута (через `opts` в `add-route!`).
- **Конфликты:** При регистрации маршрута с уже существующим методом и путем `reitit` по умолчанию перезапишет старый обработчик. Предложенная реализация `add-route!` с `merge` следует этому поведению. Это ожидаемое поведение для IoC-контейнера.
- **Документация:** Каждая публичная функция должна быть снабжена `docstring`.
- **Тесты:** Покрыть тестами конкурентную регистрацию маршрутов из разных потоков.
